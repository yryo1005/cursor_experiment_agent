# 要件定義書：機械学習実験自動更新マルチエージェントAgent (LangGraph × Gemini API)

本書は，過去の実験指示書（`order_{k}.md`）および実験結果報告書（`report_{k}.md`）の全履歴をコンテキストとして受け取り，次に実行すべき最適な実験指示書（`order_{k+1}.md`）を自動生成するマルチエージェントシステムの要件を定義するものである．

機械学習のプログラムは `@cursor_template/root_prompt.md` の指示に基づき作成される．
システム開発にあたっては，ルートディレクトリに配置されたサンプルのオーダーやレポートを検証用データ（サンプル）として使用し，プログラムのデバッグを行うこと．

---

## 1. システム概要

本システムは，LangGraphを活用したマルチエージェント・ディベート（MAD）**および**レッドキーピング（Red-Teaming / 敵対的レビュー）機構を搭載したAI Agentシステムである．
単一のLLMによる思い込みや見落とし（ハルシネーション）を防ぐため，異なる専門性を持つ複数のエージェントがディベートを行い，さらに独立したレビューエージェントが成果物を厳しく批判・検証することで，Cursor等でそのまま実行可能かつ高精度な次期実験指示書（`order_{k+1}.md`）を自律的に生成する．
テキストの生成および分析の基盤LLMには **Gemini API** を使用する．

### 1.1. 運用・配置形態（サブモジュール制約）

本システムのリポジトリは，最終的に**メインの実験リポジトリ内のGitサブモジュール（Git Submodule）として追加され，実験リポジトリのルートまたは指定ディレクトリから呼び出されて実行される．** そのため，スクリプト内の相対パスの解決やファイルの入出力先は，単体での動作ではなく「親リポジトリから実行されること」を前提に設計されなければならない．

---



## 2. 機能要件



### 2.1. コンテキスト読み込み機能 (Input Management)

- **履歴ファイルの自動検出:** 指定されたディレクトリ内から，既存の全ての `order_{k}.md` および `report_{k}.md` （$k = 1, 2, \dots, n$）を昇順で自動的に検出し，読み込むこと．
- **時系列の保持:** 実験の変遷（どのような仮説を立て，どのような結果が出て，次にどう修正したか）を各エージェントが正確に把握できるよう，時系列順にコンテキストを構築すること．
- **親リポジトリ対応のパス解決:** サブモジュール内から実行された場合でも，親リポジトリ（実験リポジトリ）のルートにある `order_{k}.md` や `report_{k}.md` の配置ディレクトリを正確に参照できるパス引数（引数によるベースパスの指定など）をサポートすること
- `run_agent.py`に実行機能を集約し，`order_{k}.md` や `report_{k}.md` が格納されたディレクトリを引数にすると良い



### 2.2. 認証・APIキー管理機能 (Authentication)

- **外部設定ファイルの参照:** Gemini APIの認証に必要なAPIトークンは，環境変数ではなくローカルファイル `tokens.json` から読み込むこと．
- **キーの特定:** `tokens.json` 内の `["gemini"]` キー（文字列）に格納されている無料プランのトークンを動的に取得し，LangChainのChatGemini（`langchain_google_genai`）等の初期化時に設定すること．



### 2.3. マルチエージェント・ディベート機能 (MAD: Multi-Agent Debate)

実験履歴の分析および次期仮説の立案において，以下の2角の専門性を持つエージェント間で議論（最大3ラウンド程度）を行わせ，合意形成（Consensus）を図ること．

- **エージェントA（データサイエンティスト）:** 主に「モデルの精度，過学習の抑制，評価指標（Metrics）の推移，高度なアルゴリズム選定」の観点から過去の report を分析し，次期の仮説を提案する．
- **エージェントB（機械学習エンジニア / インフラ）:** 主に「計算コスト（学習時間），特徴量エンジニアリングの容易さ，データの不均衡性，パイプラインのシンプルさ，実装の現実性」の観点からエージェントAの提案を評価・補正する．



### 2.4. 敵対的レビュー機能 (Red-Teaming / Critic)

- **指示書の脆弱性検知:** 生成された `order_{k+1}.md` に対し，ディベートに参加していない独立した **「レッドチーム・エージェント（批評家）」** が介入し，以下の観点で厳しく批判（ダメ出し）を行うこと．
- 前回の失敗原因（ボトルネック）が，今回の指示書で本当に解消される論理になっているか．
- 指示内容が抽象的すぎず，Cursor（AI）が迷わずにコードへ落とし込める具体性を持っているか．
- **差し戻しループ:** レッドチームから重大な欠陥を指摘された場合，システムは生成プロセスへと処理を差し戻し（最大N回），指示書を修正・アップデートさせること．



### 2.5. 指示書出力機能 (Output Generation)

- **テンプレート準拠の出力:** 最終的に承認された `order_{k+1}.md` は，`cursor_template` のフォーマット（目的，タスク詳細，制約条件，期待される出力形式など）に厳格に準拠すること．
- **出力先の親リポジトリ連携:** 生成された指示書は，サブモジュール内ではなく，呼び出し元である親リポジトリの適切な実験管理ディレクトリへ直接書き込まれるか，標準出力等で呼び出し側に安全に返却されること．
- 各エージェントの出力を`log_{k+1}.md`にまとめ出力すること．

## 3. 技術要件（LangChain / LangGraphアーキテクチャ）



### 3.1. 使用LLMおよびライブラリ

- **LLM:** Gemini API (例: `gemini-2.5-flash` または `gemini-3.5-flash`)
- 無料プランのレートリミット（RPM/TPM）を考慮し，複数エージェントの同時並行呼び出しや連続呼び出しの間に，適切なスリープ（遅延）やリトライメカニズムを実装すること．
- **フレームワーク:** LangChain (`langchain`, `langchain-google-genai`), LangGraph (`langgraph`)



### 3.2. グラフ構造（Stateの管理）

LangGraphを用いて以下の状態遷移（マルチエージェント・ワークフロー）を構築すること．

1. **[START]** $\rightarrow$ 全履歴を読み込み `State` に格納．
2. **[Node: Debate]** $\rightarrow$ エージェントAとBが履歴をもとにディベートを行い，最適な「次期実験プラン」を合意．
3. **[Node: Generate]** $\rightarrow$ 合意されたプランから `order_{k+1}.md` のドラフトを生成．
4. **[Node: Red-Teaming]** $\rightarrow$ レッドチームがドラフトを批評．
5. **[Conditional Edge: Quality Check]** $\rightarrow$ レビュー結果が「合格」なら **[END]** へ．「不合格かつ制限回数未満」なら **[Node: Generate]** へ修正指示と共に差し戻す．



### 3.3. コンテキストウィンドウ・履歴管理

- Geminiの広大なコンテキストウィンドウを活かし，ディベートの会話ログおよび全実験履歴をシステムメッセージまたは履歴として保持するが，無料プランのトークン消費量を節約するため，ディベートの各ラウンドの要約（Summary）を `State` に適切にパースして引き継ぐ設計を推奨する．

---



## 4. 入出力データ仕様



### 4.1. 入力データ

- `order_1.md` 〜 `order_{n}.md` （親リポジトリ側リソース）
- `report_1.md` 〜 `report_{n}.md` （親リポジトリ側リソース）
- `tokens.json` （`{"gemini": "YOUR_API_KEY"}` の形式）



### 4.2. 出力データ

- ファイル名: `order_{n+1}.md`
- ファイル形式: Markdown（Cursorのシステムプロンプトとして最適化され，レッドチームの検証を通過した形式．親リポジトリの実験ディレクトリへ書き出し）
- エージェントのの出力: `log_{n+1}.md`

---



## 5. 非機能要件・制約事項

- **配置の疎結合化:** 本システム（サブモジュール側）のコード内に，親リポジトリの特定のディレクトリ名をハードコーディングしないこと．必ず実行時の引数（CLI引数や環境変数，設定ファイル）を介して親のパス情報を動的に受け取れる設計にすること．
- **議論の収束制約:** エージェント同士の議論やレッドチームからの差し戻しが無限ループに陥るのを防ぐため，最大ループ回数（例：Max 3回）のカウンターを `State` で管理し，上限に達した場合は強制的にその時点の最適解を出力すること．
- **フォーマットの堅牢性:** 生成されたMarkdownが破損（コードブロックの閉じ忘れなど）しないよう，Pydantic等を用いた構造化出力（Structured Outputs）の仕組みを導入し，フォーマットエラーによる差し戻しも自動で行えるようにすること．
